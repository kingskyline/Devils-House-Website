🔵 Alpha v0.010 Architecture

Trucks & Automation

A logistics + automation + tycoon-style simulation game with modular UI systems and tech-driven economy progression.

Core Vision

Built from the ground up as a fully modular logistics automation simulator. The end goal is to handle large-scale production chains—from factories to storage to transport and sales—all driven by a robust tech-driven economy and fully UI-driven management systems.

Primary Design Goals

  • Multidimensional grid architecture
  • Modular "Data Card" resource framework
  • Automated state-based logistics
  • Scalable building & truck progression

Current Progress

Executing the Alpha v0.010 Ultimate Architecture Roadmap. Core engine decoupling is moving at full speed—Phases 1 through 4 (Boot Lock, Central Database, Event Bus, and UI State Machine) are now 100% complete. Focus has shifted to Phase 5: UI Modularisation.

Ultimate Architecture Roadmap

Current Roadmap Lifecycle — Alpha v0.010

Phase 1 — Boot System, Singleton Armor & Load Order Lock

✓ Complete

Goal: Make the entire game impossible to break on startup.

Fixes resource initialization timing, duplicate manager conflicts, random null references, and unsafe scene transitions. Enforces a strict initialization contract across BootManager, SceneBootManager, and safe lifecycle standards (Awake → Setup / Start → Link / Init → Explicit Activation).

Phase 2 — Central Game Database + Data Layer Separation

✓ Complete

Goal: Remove ALL direct Resources.LoadAll usage from gameplay/UI systems.

Introduces GameDatabaseManager as the single source of truth for items, buildings, maps, and fleet metrics. Separates static configuration data (ScriptableObjects) from runtime active instances and serialized save structs.

Phase 3 — Event System (Remove Update Polling)

✓ Complete

Goal: Kill all unnecessary Update() UI refresh logic.

Deploys a lightweight centralized GameEventBus to broadcast state actions globally, removing per-frame component query loops across layout panels.

Phase 4 — UI State Machine + Input Routing System

✓ Complete

Goal: Replace ALL panel toggle chaos with one controlled state framework.

Implements an authoritative state enum architecture alongside an interactive central Escape handler layer to manage overlapping panels smoothly.

Phase 5 — UI Modularisation (Remove God Controllers)

⚡ In Progress

Goal: Split UI_MenuController + MasterMenuController into real isolated sub-systems.

Breaks down large generic scripts into structured controller arrays across individual HUD, menu wrappers, and dedicated logic data adapters.

Phase 6 — UI Pooling + Performance Layer

○ Planned

Goal: Eliminate UI lag spikes and runtime garbage collection hitches.

Features a dedicated UIObjectPool layout to switch expensive engine mutations over to safe, structured data-rebinding streams instead of direct instantiate/destroy logic loops.

Phase 7 — Save/Load Pipeline Overhaul

○ Planned

Goal: Make save/load operations completely deterministic and staged.

Builds sequenced persistence stages to securely parse global profiles, local catalogs, world instances, and active simulation tracking metrics into secure buffer arrays without partial world corruption risks.

Phase 8 — Inspector System Standardisation

○ Planned

Goal: Unify all inspector UIs into a single lifecycle + event-driven pattern.

Cleans up hybrid structural methods across Warehouse, Converter, and ManufacturingPlant panels by enforcing strict bind-driven cycles that completely remove old Update-based refresh functions.

Phase 9 — Inspector UI Component Breakdown

○ Planned

Goal: Break inspectors into reusable UI components instead of monolithic controllers.

Transforms monolithic interfaces into lightweight modular shells referencing separate individual components for upgrade tabs, inventory bars, and production progress graphics.

Phase 10 — UI Refresh Optimisation

○ Planned

Goal: Stop destroying and recreating UI elements repeatedly.

Overhauls list generators to refresh active row metrics and values dynamically inside memory pools instead of systematically calling runtime hierarchy deletion loops on panel opens.

Phase 11 — Inspector Event Hub

○ Planned

Goal: Centralise all inspector refresh triggers into one global system wrapper.

Deploys an automated InspectorEventHub layer to unify external updates (financial shifts, warehouse capacity actions) down into singular refresh request distributions.

Phase 12 — UI State Cleanup for Inspectors

○ Planned

Goal: Prevent overlapping inspectors and ghost UI states.

Establishes an authoritative InspectorStateManager to enforce singular interaction panels, guarantee clean layout unbinding routes, and safely clear old state metrics.

Phase 13 — UI Safety + Performance Stability Pass

○ Planned

Goal: Remove hidden performance risks from core inspector components.

Switches active rendering paths to event-driven triggers, introduces safe zero-capacity division overrides inside progress slider calculations, and blocks redundant layout reconstruction actions under high logistics loads.

Phase 14 — UI Event System Hardening

○ Planned

Goal: Eliminate event leakage risks and strengthen subscription lifecycle pathways.

Locks down unbind execution steps across structural warning indicators and inspector views, ensuring elements safely drop listener references to remove stale dictionary indexing bugs.

Phase 15 — Menu + Save System Safety Hardening

○ Planned

Goal: Prevent save corruption and unsafe file operations during menu transitions.

Adds explicit exception Handling strategies around manual profile deletions, wraps loading card metadata loops against corrupted JSON arrays, and unifies load selections safely.

Phase 16 — Gameplay UI Economy Integration Hardening

○ Planned

Goal: Stabilise economy-linked UI systems and prevent hidden dependency failures.

Introduces context checking parameters inside global progression instances, prepares layout paths to support upcoming data-driven scaling metrics, and caches LINQ collection lookups during market board transactions.
